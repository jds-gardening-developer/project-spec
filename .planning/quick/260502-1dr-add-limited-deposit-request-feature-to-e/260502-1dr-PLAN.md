**Edit 1 — three schema rows (insert above the empty `|  |  |  |` row at L530):**

```
| deposit\_amount | decimal (nullable) | the amount the customer was asked to pay to lock the order in. Defaults to 50% of order subtotal (ex VAT) when admin clicks "Request Deposit"; admin can override. Null when no deposit was taken. |
| deposit\_requested\_at | datetime (nullable) | when the deposit request was sent to the customer |
| deposit\_received\_at | datetime (nullable) | when the deposit cash landed (Stripe deposit-kind webhook OR manual mark by Finance for bank transfers) |
```

**Edit 2 — full replacement for the L562 paragraph (replaces the entire single-line paragraph that starts "Deposits \- MacPlants does not take deposits..."):**

```
**Deposits.** For long-lead-time orders (typically 6+ months between commitment and delivery), MacPlants may take a deposit to lock stock in. The deposit is purely a customer-facing payment request — it is **not** a Xero entity, and it does **not** create a separate Invoice in the ERP. It exists as three fields on the Order (`deposit\_amount`, `deposit\_requested\_at`, `deposit\_received\_at`) plus a notification document sent to the customer.

* On any Approved or Confirmed order, an admin can click "Request Deposit". A modal pre-fills 50% of the order subtotal (ex VAT) — admin can override to any amount.
* The system emails the customer a Deposit Request document with a Stripe payment link. The order's `deposit\_amount` and `deposit\_requested\_at` fields are populated.
* Customer pays the deposit (Stripe link or bank transfer). When the cash lands, `deposit\_received\_at` is populated — automatically via Stripe webhook for Stripe payments, or manually marked by Finance for bank transfers.
* Stock reservation is unchanged: it follows the order's Confirmed status as today (see Order Phase below). Taking a deposit does not gate reservation; admins move long-lead orders into Confirmed at the point the customer commits, and the deposit request runs in parallel.
* The deposit cash is held by Finance in a holding account ("Other Debtors" in Xero) until the real Invoice posts at dispatch. There is no Xero invoice for the deposit itself.
* At dispatch, the standard Invoice for the full order amount is generated and posts to Xero per the existing Xero Posting Flow. The customer-facing rendering of this invoice shows: Total / Deposit Paid / Balance Due. Customer pays the balance; Finance reconciles the holding-account deposit and the new balance payment against the single Xero invoice and marks the ERP Invoice as Paid.
* If the order's value changes after the deposit is paid, the deposit amount stays as it was. The customer's Balance Due at dispatch adjusts accordingly. If the order shrinks below the deposit amount, the difference is refunded via the standard refund / credit note flow (PRD-3.3).

**Order Paid status under the deposit flow.** The existing rule is unchanged: an order is Paid when its active Invoice has status Paid. The deposit fields do not enter the calculation. Between deposit-received and dispatch, the order is still Unpaid (the active full-amount Invoice has not been marked Paid). The order list / dashboard surfaces a derived "Deposit Received" state for these orders so admin can see at a glance that long-lead orders have their deposit in.

**Stripe link routing.** Stripe Payment Links are issued for two distinct purposes: (a) paying an Invoice (existing flow above), and (b) paying a Deposit Request. Each link carries a metadata field on the Stripe object identifying which it is. The webhook handler routes on this metadata: an Invoice-kind payment marks the Invoice Paid; a Deposit-kind payment populates `deposit\_received\_at` on the Order and **does not touch any Invoice**. The two flows must never share code paths.
```
