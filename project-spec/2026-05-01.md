  #### **Purpose**

To detect deliberate or careless mishandling of stock by staff. This is a narrow-scope, Super-Admin-only investigation tool — not a general audit trail.

#### **What It Must Do**

##### **Scope of Logged Actions**

The log captures stock-affecting actions only:

* Quantity adjustments on a batch (up or down)  
* Batch status changes (e.g. Growing → Trade Ready)  
* Batch location changes (e.g. Bed 1.3 → Bed 2.7)  
* Batch deletions

Order edits, PO edits, customer edits, quote edits, login events, and any non-stock admin action are explicitly out of scope and must not be written to this log.

Every logged entry is generated automatically by the system at the point the action is committed. Staff cannot edit, delete, or backdate log entries from any interface.

##### **Logs Screen (Super Admin only)**

A "Logs" menu item appears in the left-hand navigation. It is visible only to users with the Super Admin role. Admin and Staff users do not see this menu item or have access to the screen by direct URL.

The screen displays log entries in a paginated table, newest first, with the following columns: Timestamp, Staff Member, Action Type, Batch (linked to batch detail screen), SKU, Before → After, Source (PWA / ERP), Linked Shrinkage (linked to shrinkage record where applicable).

##### **Filters**

* Date range (from / to date pickers, plus quick filters: Last 7 days, Last 30 days, Last 3 months, Last 12 months)  
* Staff member (dropdown, multi-select)  
* Action type (multi-select: quantity\_adjust, status\_change, location\_change, batch\_deleted)  
* Source (PWA / ERP / both)  
* SKU (free-text search)

Default view on screen load: Last 30 days, all staff, all action types, all sources.

##### **Summary Panel**

A summary panel sits above the table and reflects whatever filters are currently applied. For each staff member appearing in the filtered results, it shows: total actions, total units adjusted down, total units adjusted up, total status changes, total location changes, total batch deletions.

This is the primary tool for spotting patterns. The table beneath is for verifying what the summary suggests.

##### **Retention**

Log entries are retained for 3 months from creation. A scheduled job (daily) deletes entries older than 3 months. No archival to cold storage; older entries are permanently removed. (Automatic)

##### **Performance**

The table must be indexed on `user`, `timestamp`, `plant_batch`, and `action_type` to support fast filtering at scale. Queries must be server-side (filters applied at the database level, not in the browser).

#### **How We Know It's Done**

* A Super Admin user lands on the Logs screen and sees stock-affecting actions from the last 30 days, newest first, with the summary panel populated.  
* Admin and Staff users cannot see the Logs menu item and receive a 403 if they navigate to the URL directly.  
* Filtering by date range, staff member, action type, source, and SKU updates both the table and the summary panel correctly.  
* Each row links through to the affected batch and (where applicable) the linked shrinkage record.  
* CSV export downloads the full filtered result set, not just the current page.  
* Stock actions performed on the PWA appear in the log within seconds of being committed.  
* Stock actions performed in the ERP appear in the log within seconds of being committed.  
* Order edits, PO edits, customer edits, and login events do NOT appear in the log.  
* Entries older than 12 months are automatically removed by the cleanup job.  
  ---

A few notes on what I locked down so the devs don't have to ask:

* **Denormalised SKU on the log entry.** If a plant variant's SKU is ever edited (rare, but possible), the log entry preserves the SKU as it was at the time of the action. This matters for investigation integrity — you don't want a renamed SKU to obscure what was actually adjusted.  
* **`before_value` and `after_value` as strings.** Quantity, status, and location are different types, but storing them as strings on the log row keeps the schema simple. The action\_type tells you how to interpret them.  
* **Cleanup job is delete, not archive.** You explicitly said the purpose is narrow and the retention can be tight, so no archive table. Saves complexity.  
* **Server-side filtering is in the spec.** Without this, the devs might build it client-side, which would be fine for the first month and painful by month six.  
* **Permission enforcement is explicitly tested.** The "How We Know It's Done" includes the 403 check on direct URL access — this is the kind of thing that gets missed in informal QA.

One small thing worth raising with the devs at kickoff: confirm they're writing to this log inside the same database transaction as the stock change itself. If the log write is a separate async call, you can lose entries when something fails mid-action — which is exactly the case where you'd most want the log. Should be a one-line confirmation, but worth nailing down.

## **What Is It**

The foundation of the entire system. A single database that holds every plant variety Macplants grows or sells, every pot size, every price, and every batch of stock — whether it's available, growing, on order from a supplier, or reserved for a customer. This replaces GrowMaster as the source of truth for stock.

## **What It Must Do**

### **Plants**

* In Settings \> Plants. List all of the plants in the database. This will detail Plant Name, Plant Sizes, Available on WebShop Boolean, VAT Applicable boolean, Special Plant Boolean.

* User can click on plant, view all different pot sizes, cost price, trade price and retail price. This works the same as on the plant table.

* User will be able to toggle each pot size as ON or OFF for the Retail Webshop available. User will see the plant category as a pill.

* Edit icon on the right hand side. User clicks on this and then they can see the plant name, description, care instructions, can edit prices & dimensions (text field) for each pot size for cost, trade or retail.

* In Settings \> Plant Settings. The user can see all plants that have the boolean 'Special Plant: True'. The User can add or remove any plant from this list easily. This boolean flag applies to all of the POT Sizes of that plant.

* In Settings \> Plant Settings. Use can set the VAT as 20%, 0% or NO VAT, this is a dropdown on each plant & will apply to all batches of that plant

* In Settings Plants \> Add Plant — Admin can Create a New Plant which is added to the database, this will automatically generate the SKU.  User will be prompted to full out Plant Name, description etc & all prices of different plant pot sizes when creating a new plant. 

* Each Plant supports multiple pot sizes per plant (7cm, 9cm, 1L, 2L, 2LD, 3L), each with its own SKU, retail price, trade price, cost price, and dimensions.

### **Plant Batches**

* Status — Growing, Reserved, Trade Ready, Retail Ready.

* Track inventory through batches — each batch is a group of plants of the same variety and size moving through the same lifecycle.

* Allow admin to create batches at any time without needing a purchase order from ERP. This is a core daily action on the ERP.

* Track batch status: growing, trade ready, reserved (committed to an order), retail-ready.

* Allow admin/staff user to add / remove / update quantity of batches.

* Provide a search that lets staff find any plant by latin name and instantly see the full stock picture.

* Every stock change — status updates, quantity adjustments, wastage, new batches — must be recorded with who did it and when. This can be viewed from the batch line and also from a central admin panel which ONLY the super admin can see. This will be a side panel that says 'Updates', it will have a table of all batch updates detailing 'Staff Member, update, date, time'.

* Add override "Unavailable On Webshop" — true/false boolean that can be toggled on the ERP. A plant pot size can be retail-ready status but NOT available on the webshop (e.g. too tall to ship at certain times of year). This is about shippability, not quality.

* Each batch has a "ready date." When the ready date arrives, the system auto-creates a TO DO task for staff to physically check the batch. The status does NOT change automatically — staff verify and then manually update if ready. This is important because readiness is variable (sunshine, weather etc).

* **Shrinkage** — When staff adjust a batch quantity downward (e.g. from PWA or ERP), the system must prompt for a reason (e.g. disease, dead, weeds, stock take adjustment). The system calculates the difference automatically. This captures shrinkage data from day one for future reporting. The alert window will have 2 buttons: 'Record Shrinkage' (user must provide a reason from the dropdown, can select Other with Comment) and 'Don't Record'.

* Every plant could potentially be available in every pot size (7cm, 9cm, 1L, 2L, 3L, etc.). The system must allow any plant to be configured with any size, even if in practice only certain combinations are used.

* **Locations & digital map:** Every batch has a location which references a bed/area on the nursery digital map. The map is being updated to add two further sections to the existing layout — including the area between the twin-span tunnels and the shade tunnel, known as Glenkint (code "GK"). One of the two new sections falls just outside the current map outline so the map will be redrawn larger. Beds will be subdivided into 10 sub-sections (e.g. bed 1.1 through 1.10) rather than the previous N/S/E/W quarters — this dramatically reduces the search time for staff finding a specific batch on a long bed, and lets multiple people move stock around without losing track. **Jake to supply the updated map for upload.**

* **Initial stock take is required at go-live.** GrowMaster's stock figures are not trusted — the system reports \~160k–170k plants, the real figure is significantly lower. From the date of import onwards, the source of truth is the ERP; any new stock comes through a PO and any reduction comes through an order, shrinkage, or stock take. Prices, however, will be carried forward from the EPOS back-office (which already has cost and retail prices cleaned up — \~2,900 product entries).

## **What Is It**

A structured way to order stock from external suppliers, track what's been ordered vs what's actually arrived, and update inventory when shipments come in. Replaces the current process of receiving invoices and manually uploading them.

Admin User will RECONCILE the stock when it arrives. MacPlants staff will note actual quantities which arrived on a piece of paper and this data will be INPUT into the system using the Purchase Order which was created.

## **What It Must Do**

* In Settings \> Suppliers. Maintain a supplier database with name, contact, email, phone, and address. (A Supplier can be a customer as well — however these are not linked, separate accounts.)

* One PO equals one supplier — you cannot split a PO across multiple suppliers.

* User is able to easily duplicate a PO — they can then assign the duplicate to a different supplier.

* Staff / Admin User can add Notes to PO at ANY stage on ERP.

* Auto-generate a unique PO reference number for each order.

* Send the PO directly to the supplier by email from within the ERP, with a formatted PDF Doc attached.

* User can create Tasks from PO. User can see all tasks related to PO on the single PO Screen.

* When stock arrives, staff RECONCILE actual quantities received per line. The received quantity may differ from what was ordered — this is normal.

* When Admin reconciles PO on the ERP, they will add the batches, quantity, batch stage (Growing, Retail Ready, Trade Ready), Ready Date (optional) and Location. All controlled from ERP.

* **Unit cost handling on PO:** Each PO line carries a unit cost (pulled from the plant's cost price by default). Editing unit cost on a PO line changes it for that PO only. To change the cost price across all future POs for that plant, the cost price must be edited on the plant record itself (Settings → Plants). The PO reconcile screen is for reconciling location, quantity and batch status — not for changing unit cost; unit cost is fixed at the point the PO was sent.

* On purchase-orders all screen — Display all open POs, their status and expected delivery dates. Pills detail how many of that status there are.

* Lock completed POs so they cannot be edited after they have been reconciled.

* Show Stepper on top of individual PO to see what stage it is at.

* Staff can CRUD a PO, add line items, add notes, and save as draft.

* Staff can view 'progress' of individual PO — progress stepper at top of screen.

* Once a PO is approved it cannot be updated or deleted. Only cancelled.

* Staff can send a PO to a supplier via email from within the ERP — this will have Supplier Order PDF attached. Staff can download Supplier Order PDF.

**Deposits.** For long-lead-time orders (typically 6+ months between commitment and delivery), MacPlants may take a deposit to lock stock in. The deposit is purely a customer-facing payment request — it is **not** a Xero entity, and it does **not** create a separate Invoice in the ERP. It exists as three fields on the Order (`deposit\_amount`, `deposit\_requested\_at`, `deposit\_received\_at`) plus a notification document sent to the customer.

* On any Approved or Confirmed order, an admin can click "Request Deposit". A modal pre-fills 50% of the order subtotal (ex VAT) — admin can override to any amount.
* The system emails the customer a Deposit Request document with a Stripe payment link. The order's `deposit\_amount` and `deposit\_requested\_at` fields are populated.
* Customer pays the deposit (Stripe link or bank transfer). When the cash lands, `deposit\_received\_at` is populated — automatically via Stripe webhook for Stripe payments, or manually marked by Finance for bank transfers.
* Stock reservation is unchanged: it follows the order's Confirmed status as today (see Order Phase below). Taking a deposit does not gate reservation; admins move long-lead orders into Confirmed at the point the customer commits, and the deposit request runs in parallel.
* The deposit cash is held by Finance in a holding account ("Other Debtors" in Xero) until the real Invoice posts at dispatch. There is no Xero invoice for the deposit itself.
* At dispatch, the standard Invoice for the full order amount is generated and posts to Xero per the existing Xero Posting Flow. The customer-facing rendering of this invoice shows: Total / Deposit Paid / Balance Due. Customer pays the balance; Finance reconciles the holding-account deposit and the new balance payment against the single Xero invoice and marks the ERP Invoice as Paid.
* If the order's value changes after the deposit is paid, the deposit amount stays as it was. The customer's Balance Due at dispatch adjusts accordingly. If the order shrinks below the deposit amount, the difference is refunded via the standard refund / credit note flow (PRD-3.3).

**Order Paid status under the deposit flow.** The existing rule is unchanged: an order is Paid when its active Invoice has status Paid. The deposit fields do not enter the calculation. Between deposit-received and dispatch, the order is still Unpaid (the active full-amount Invoice has not been marked Paid). The order list / dashboard surfaces a derived "Deposit Received" state for these orders so admin can see at a glance that long-lead orders have their deposit in.

**Stripe link routing.** Stripe Payment Links are issued for two distinct purposes: (a) paying an Invoice (existing flow above), and (b) paying a Deposit Request. Each link carries a metadata field on the Stripe object identifying which it is. The webhook handler routes on this metadata: an Invoice-kind payment marks the Invoice Paid; a Deposit-kind payment populates `deposit\_received\_at` on the Order and **does not touch any Invoice**. The two flows must never share code paths.

# **PRD-3.4: Trade Shows (Stock Reservation & Sales Reconciliation)**

NB \- Tech Team to Ignore this \- no new functionality here \- this is just to document how this will be handled

## **Data Model**

*No new objects. Trade shows reuse the existing Order (with custom Title) and Purchase Order (for unsold-stock returns) primitives. The only show-specific behaviour is suppressing customer-facing notifications when the linked customer is the dummy show customer.*

## **What Is It**

Trade shows (Scone Palace, Powis, RHS shows, Chelsea, farmers' markets, etc.) are a meaningful part of the Macplants business — Scone alone takes \~4 weeks of one person's time to prepare. The ERP must support reserving stock for a show, generating a pick list, taking the show stock out of online/trade availability, and reconciling sales after the show closes.

There is no bespoke "show" module. Trade shows are handled entirely with two existing primitives — a normal Order (with a custom title) and a normal Purchase Order — used in sequence. This keeps the codebase clean and means show data flows through the same reporting, invoicing, and audit machinery as everything else.

## **What It Must Do**

* **Step 1 — Create a titled order to reserve show stock.** Admin creates a normal order, sets the Order Title (PRD-03) to something descriptive (e.g. "Scone 2026"), and adds the plants and quantities being taken to the show as line items. The order can sit against a dummy show customer (or any customer admin chooses to use for shows). Creating this order reserves the stock in the standard way, so it immediately disappears from the trade portal and the WordPress retail site. The pick list for the show is generated from this order exactly like any other order — no separate show pick-list flow.

* **Step 2 — After the show, edit the order to reflect what actually sold.** When the team returns from the show, admin edits the line item quantities on the original order down to the quantities that were actually sold. The order is then pushed through the standard invoice flow against the dummy show customer. This is what captures the show's revenue: the invoice value equals the actual takings, and the sales appear in Xero / dashboards as one consolidated "show invoice". Payment against the invoice is recorded manually as a mix of card terminal, PayPal, and cash takings — Jake reconciles these against the single invoice. Cash banking sits with Jake and is out of scope for the ERP.

* **Step 3 — Create a Purchase Order to bring unsold stock back in.** The unsold plants returning from the show are added back into stock via a normal PO. Admin does a quick stock-take of what came back in the van, then creates a PO listing each plant and the returning quantity. Reconciling that PO puts the plants back into the correct nursery location with the right batch status. This PO is the formal record of incoming stock from the show — and, by comparison with the original order, gives the audit trail of what actually sold (taken − returned \= sold).

* **Why two records (order edit \+ return PO) instead of one auto-restocking edit:** Editing the order down does NOT auto-return stock. The return PO is the explicit, auditable mechanism for stock coming back in. This mirrors how every other inbound stock movement works in the system and avoids hidden side-effects on the order screen.

* **No customer-facing notifications.** The dummy show customer must not receive emails. Notifications should be suppressed for show-flagged orders.

* **Year-on-year visibility falls out for free.** Because each show is one titled order plus one return PO, Macplants can look back across years and see what was taken, what sold, and what came back, without any new reporting code.

* Shows include farmers' markets and small local events — not just the big RHS-style ones. The same pattern applies regardless of show size.

## **How We Know It's Done**

* Admin can create an order with a custom title (e.g. "Scone 2026") that reserves the stock and removes it from the trade portal and website.

* A show pick list downloads as a PDF in the same format as any normal order pick list.

* After the show, admin can edit the order quantities down to actual sold quantities and push the order through the standard invoice flow. Payment is recorded manually with a mixed-tender note (terminal / PayPal / cash).

* Admin can then create a separate PO for the returning unsold stock; reconciling that PO puts the plants back into the correct nursery location.

* Show orders never trigger customer-facing emails.

* For any past show, Macplants can pull the original order, the final invoice, and the return PO and see the full taken / sold / returned picture.

## **5.4 Trade Customer Portal**

Login URL: /.trade.macplants.co.uk/login Integration: Stripe

## **What Is It**

A self-service web portal where trade customers log in, search for plants and pricing, build orders, select delivery methods, advise timelines, and track their order status. This replaces the current process where customers email a wish list, staff manually search GrowMaster, and a spreadsheet is sent back.

## **What It Must Do**

### **Trade Customer Portal**

* Trade customers log in with credentials created by Macplants staff. No self-registration.

* Customers can search plants by name.

* All plants are shown on the portal unless marked as "unavailable." Even plants that are currently out of stock are visible — Macplants can always reject the order. Plants marked as unavailable are hidden entirely.

* Retail-ready stock is excluded from the trade portal. Trade customers can ONLY buy retail-ready plants via the shop or mail order.

* Stock quantities ARE shown on the trade portal (earlier concern about hiding quantities for "special" plants was dropped — of \~1,200 plants, only \~20 are truly special. These will be flagged via internal notes rather than hiding quantities across the board).

* Customers can build a quote/order with plant lines, quantities, a desired delivery date (this could be 6 months in advance), and a fulfilment method.

* Two purchase paths:

  * **Quote path** (default for most jobs, especially when pricing up a job for an end client): customer submits a request → "Awaiting Response" → Macplants reviews/substitutes → customer accepts line-by-line → order.

  * **Instant buy path** (for known customers who just need plants quickly): customer adds to basket, and if within their credit limit the order flows straight to Confirmed. If the basket exceeds the credit limit, a Stripe payment screen appears so they can pay the balance over the limit upfront and continue. This removes friction for repeat trade customers who don't need a full quote process for routine orders.

* Running totals show: line totals, subtotal (ex VAT), VAT amount, grand total (inc VAT). Deposits are not taken at the portal-checkout step — they are an admin-side action on a Confirmed order (see PRD-3.1).

* Every order submitted via the Quote path goes to "Awaiting Response" status. Instant-buy orders go straight to Confirmed.

* Customers can Accept quotes sent back by Macplants staff.

* When Admin Updates Quote for trade customer, they can trigger an email to tell the customer to check their account as there has been an update.

* Customers can see the status of all their orders at every stage.

* **IMPORTANT:** Customer can accept the quote line by line (not ALL items), if this has been approved by the MacPlants staff.

* Customers can view their order history.

* Collection dates selectable by trade customer when ordering. If an order is placed within the same day, earliest collection is 48 hours later (for pick/pack time).

### **Internal Quote Builder (Admin Side)**

* Everything the portal offers, plus full stock visibility on what is Available.

* See cost price of each plant size for MacPlants staff. ESSENTIAL. Automatically calculate the profit margin percentage.

* Override pricing on any line manually.

* Add internal notes (not visible to customer) and customer-visible notes per QUOTE — ESSENTIAL. MacPlants may want to discuss any swaps that happened.

* Send quotes to customers with one click.

* **Substitution workflow (IMPORTANT):** When reviewing a customer's quote request, staff can strike through (DELETE) unavailable lines and add substitution lines at the bottom with a note and unit price. The amended quote is sent back to the customer. The customer can then accept or reject each line individually (including substitutions) and submit. This eliminates the back-and-forth that currently wastes significant time. Once the customer submits their selections, it's a closed deal — no further manual work needed.

* **Per-line notes (not a single quote-level note):** Notes for substitutions and line-specific comments must be attached to the individual line item, NOT a single note at the bottom of the quote. On the generated PDF/quote view, the note appears directly underneath the affected line (e.g. "substituted for Crocosmia Lucifer"). A single bottom-of-quote note was explicitly rejected as too confusing — GrowMaster's per-line approach is the model to follow.

* Substitutions will be rare in practice. Because trade customers see live availability on the portal, they cannot order plants that aren't in stock. The main remaining substitution case is forward-dated orders (e.g. plants requested for delivery in 6 months that we don't currently hold).

* A dedicated Pending Orders screen shows all orders waiting for stock. When stock becomes available, the affected order is highlighted and the manager can click Confirm to approve it.

## **How We Know It's Done**

* Trade customers can log in with ERP-created credentials.

* Portal shows searchable plants with availability and prices.

* Customers can build quotes with line items, quantities, delivery date, and fulfilment method.

* Running totals (subtotal, VAT, total) calculate correctly on the trade portal. Deposit handling is admin-side only (PRD-3.1) and does not appear in portal totals.

* Customer-submitted orders always go to Awaiting Response.

* Admin can review, modify, and send quotes back to customers.

* Customers can approve quotes in the portal — OR individual line items of quote.

* Admin has a dedicated Pending Orders screen showing orders awaiting stock.

* Admin Side — Internal quote builder shows full stock visibility.

# **PRD-7: WordPress & ERP API Contract**

## **What Is It**

The API and integration rules that connect the new WordPress retail website to the ERP. WordPress displays products and handles browsing. The ERP owns all data, pricing, stock, and order processing. Checkout happens on the ERP (subdomain), not on WordPress.

**Note:** The WordPress site is built by Paul/Craig. The ERP API is built by the ERP Team. This document defines the contract between the two so both teams build in parallel.

## **What It Must Do**

* The ERP provides API endpoints for WordPress to pull: all products with variants, images, and pricing; stock levels per SKU; new products since a given timestamp.

* WordPress syncs products into Custom Post Types with ACF fields. SKU is the linking key between systems.

* Stock sync is manual. When a new plant is added to the ERP is needs to be MANUALLY added to the WP site.  As this wont’ happen often it is the best solution.

* On ERP there is an Available / Unavailable on WebShop Override.

* If the ERP API is down, WordPress does not zero out stock. It retries and logs the failure.

* Checkout is handled on the ERP, hosted on a subdomain (e.g. checkout.macplants.co.uk). When a customer clicks checkout on WordPress, they are redirected to the ERP which validates stock, takes payment via Stripe, and creates the order.

* Customer registration on checkout page (which is on the ERP \- so will create an account int the ERP \- customer is never logged into WP)

* Customer will be able to login to WP Site Via ERP \- See Echo3 Education Flow

* API authentication uses API keys per environment with rate limiting.

* WordPress owns SEO fields (meta titles, descriptions). The ERP owns product data (names, descriptions, variants, pricing, stock). Neither system overwrites the other's fields.

* WP & ERP connects the plants with the SKU ? 

## **How We Know It's Done**

* Stock sync updates only stock fields without overwriting editorial content.

* New products can be synced on demand.

* Basket validates stock availability against the ERP before checkout.

* Checkout redirects to the ERP-hosted page with basket data.

* Customer registration on WordPress pushes data to the ERP.

* Customer account page shows order history from the ERP.

* SKU serves as the unique linking key.

* Sync is re-runnable without duplicates.
